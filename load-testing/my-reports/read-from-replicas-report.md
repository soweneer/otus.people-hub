# Переключение чтения с мастера на чтение с 2х реплик

## Условия
- разгон до 400 VU за 200с, сценарий: `GET /user/search` по рандомному имени-фамилии с имитацией скролла вниз по списку найденного и точечные `GET /user/{id}` (нагрузочный скрипт - в search.js)
- PostgreSQL 18.4 в docker
- 1 млн пользователей, GIN-индексы по name и surname с `gin_trgm_ops`

## Результаты (см файлы `run1-master-only.html` и `run2-master-plus-replicas.html`)

| Метрика | Только мастер | Мастер + 2 слейва |
|---|---|---|
| Запросы | 63 255 (303.2 rps) | 64 470 (308.7 rps) |
| 500е ошибки | 0 % | 0 % |
| `user/{id}` медиана | 161 ms | **53 ms** |
| `user/search` медиана | 539 ms | **446 ms** |
| `user/{id}` p95 | 702 ms | 555 ms |
| `user/search` p95 | 1.27 s | 1.71 s |
| `user/{id}` p99 | 953 ms | 838 ms |
| `user/search` p99 | 1.7 s | 2.54 s |

## CPU в середине прогонов

| Нода | Только мастер | Мастер + 2 слейва |
|---|---|---|
| pg-master | **2381 % CPU** | 15 % CPU |
| pg-replica-1 | 0.00 % | 1283 % CPU |
| pg-replica-2 | 0.01 % | 1064 % CPU |

Вывод: разгрузили мастер, нагрузка на чтение пошла на слейвы.

## Выводы

1. **Переключение работает.** При первом прогоне реплики простаивали, после переключения read-запросов на реплики нагрузка переключилась и равномерно размазалась по двум репликам, мастер разгрузился с 2381 % до ~15 % CPU.
2. **Медианная латентность улучшилась в разы**: `user/{id}` — 161 ms → 53 ms (×3),
   `user/search` — 539 ms → 446 ms (×1.2). Быстрые запросы перестали стоять в очереди к
   единственному узлу.
3. **Суммарная пропускная способность не выросла** (~303 → ~309 req/s), а хвост (p95/p99)
   даже удлинился. Причина — стендовое ограничение: все три узла делят CPU одной машины,
   суммарно доступные ~2400 % CPU были исчерпаны и в baseline. Репликация перераспределяет
   нагрузку, но не добавляет железа.
