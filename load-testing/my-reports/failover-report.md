# Имитация падения мастера и промоушен реплики до мастера

## Условия

- PostgreSQL 18.4: `pg-master` + 2 реплики
- Кворумная синхронная репликация: `synchronous_standby_names = ANY 1 (replica_1, replica_2)`
- Скрипт нагрузки `load-testing/write-load.js` (20 VU в течение 240 с)

## Сценарий работы

* Запускаем write-load.js
* Через 60 секунд `docker kill pg-master` (запросы на запись начинают падать, ибо живого мастера нет, счетчик writes_failed в k6 растет)
* Сравниваем LSN у обеих нод (`docker exec <container-id> psql -U postgres -tc "select pg_last_wal_receive_lsn();"`), 
* Выбираем ту у которой он больше, выполняем на ней `docker exec <container-id> psql -U postgres -tc "select pg_promote();"`
* Создаем на новом живом мастере (он же бывший pg-replica-1) слот для репликации (`select pg_create_physical_replication_slot('replica_2_slot')`)
* Единственный оставшийся слейв перенацеливаем на новый мастер через `alter system set primary_conninfo = <..настройки подключения..>` и `select pg_reload_conf()`
* Наблюдаем на графике восстановление обработки запросов на запись, и счетчик writes_confirmed снова растет
