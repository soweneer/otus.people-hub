# Сравнение: чтение с мастера vs чтение с реплик

Замеры по плану `load-testing/search.js`: ramp до 2000 VU за 200 с, сценарий = поиск
`/user/search` с пагинацией + точечные `/user/{id}`. Всего ~66–69 тыс. запросов за прогон.

## Условия

- PostgreSQL 18.4: `pg-master` + `pg-replica-1` + `pg-replica-2` (потоковая репликация, async), docker compose.
- 1 млн пользователей, GIN-индекс `(surname gin_trgm_ops, name gin_trgm_ops)`.
- Приложение: .NET, Release, локально на `:8080`; k6 в докере.
- Роутинг чтения: `NpgsqlMultiHostDataSource`, `TargetSessionAttributes.PreferStandby` для read-only запросов.
- **Baseline**: connection string содержит только мастер (`localhost:5432`).
- **Replicas**: все три хоста + `Load Balance Hosts=true`.
- ⚠️ Все узлы БД, приложение и k6 работают на одной физической машине — общий CPU один и тот же.

| Метрика | Baseline (только мастер) | Reads → реплики |
|---|---|---|
| http_reqs | 68 901 (299.5 req/s) | 65 965 (286.5 req/s) |
| ошибки HTTP | 0 % | 0 % |
| med общая | 3.31 s | **1.24 s** |
| p95 общая | 6.22 s | 15.05 s |
| `user/{id}` med | 3.17 s | **136 ms** |
| `user/{id}` p95 | 5.81 s | 14.42 s |
| `user/search` med | 3.43 s | **1.58 s** |
| `user/search` p95 | 6.31 s | 15.59 s |

## Снимок нагрузки в середине прогона (docker stats)

| Узел | Baseline | Reads → реплики |
|---|---|---|
| pg-master | **2140 % CPU** | 913 % CPU |
| pg-replica-1 | 0 % | 901 % CPU |
| pg-replica-2 | 30 % | 569 % CPU |
| подключения | мастер: `too many clients`, реплики пустые | все три узла загружены |

## Выводы

1. **Роутинг работает**: в baseline реплики простаивали (0 %), после переключения read-only
   запросов на `PreferStandby` нагрузка распределилась по всем трём узлам, мастер разгрузился
   с 2140 % до ~910 % CPU.
2. **Медианная латентность улучшилась в разы**: `user/{id}` — 3.17 s → 136 ms (×23),
   `user/search` — 3.43 s → 1.58 s (×2). Быстрые запросы перестали стоять в очереди к
   единственному узлу.
3. **Суммарная пропускная способность не выросла** (~299 → ~287 req/s), а хвост (p95/p99)
   даже удлинился. Причина — стендовое ограничение: все три узла делят CPU одной машины,
   суммарно доступные ~2400 % CPU были исчерпаны и в baseline. Репликация перераспределяет
   нагрузку, но не добавляет железа. На отдельных хостах ожидаемый эффект — рост ёмкости
   чтения примерно ×3.
4. При 2000 VU обе конфигурации перегружены (все узлы упираются в `max_connections=100`);
   для сравнения «в рабочей зоне» имеет смысл повторить замер на 300–500 VU.
