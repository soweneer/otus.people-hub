# Отчёт: репликация PostgreSQL, перенос чтения на слейвы, отказоустойчивость

Домашнее задание: настроить потоковую репликацию (1 мастер + 2 слейва), перевести
запросы на чтение на слейвы, показать перераспределение нагрузки замерами, настроить
кворумную синхронную репликацию и провести эксперимент с аварийной остановкой мастера
под write-нагрузкой.

Стенд: Windows 11 + Docker Desktop, PostgreSQL 18.4, приложение .NET (ASP.NET Core),
нагрузка — k6 в докере. Все контейнеры БД, приложение и генератор нагрузки работают на
одной физической машине (это важно для интерпретации замеров, см. раздел 4).

---

## 1. Как настроена репликация

Кластер описан в `docker-compose.yml`: контейнеры `pg-master`, `pg-replica-1`,
`pg-replica-2` в общей docker-сети `pg-cluster`, наружу проброшены порты 5432/5433/5434.

**Мастер** (`pg-master`) запускается с параметрами:

```
wal_level=replica
max_wal_senders=10
max_replication_slots=10
```

При первичной инициализации выполняется `docker/master-init.sh`:

- создаёт роль `replicator` с правом `REPLICATION`;
- создаёт два физических слота репликации (`replica_1_slot`, `replica_2_slot`) — слоты
  гарантируют, что мастер не удалит WAL, который реплика ещё не получила;
- добавляет в `pg_hba.conf` строку `host replication replicator all scram-sha-256`
  (обычные `host all ...` записи replication-подключения не покрывают);
- включает кворумную синхронную репликацию:
  `ALTER SYSTEM SET synchronous_standby_names = 'ANY 1 (replica_1, replica_2)'`.

Почему `ALTER SYSTEM`, а не параметр командной строки: `-c synchronous_standby_names=...`
действует и на временный init-сервер, который docker-entrypoint поднимает для создания
базы. Реплик в этот момент ещё не существует, кворум собрать не из кого — и первый же
`CREATE DATABASE` зависает в ожидании подтверждения навсегда (проверено на себе:
инициализация кластера с нуля намертво висла). `ALTER SYSTEM` пишет параметр в
`postgresql.auto.conf`, и он вступает в силу только при финальном старте сервера, когда
инициализация уже завершена.

**Реплики** используют кастомный entrypoint `docker/replica-init.sh`: при пустом каталоге
данных реплика снимает базовую копию с мастера и стартует в режиме hot standby:

```
pg_basebackup \
    --pgdata="$PGDATA" \
    --dbname="host=pg-master port=5432 user=replicator application_name=replica_N" \
    --slot=replica_N_slot \
    --wal-method=stream \
    --write-recovery-conf \
    --progress
```

`--write-recovery-conf` создаёт `standby.signal` и прописывает `primary_conninfo`
в `postgresql.auto.conf` — после старта реплика сама подключается к мастеру и стримит WAL.
`application_name=replica_N` в строке подключения — то имя, по которому мастер узнаёт
реплику в `synchronous_standby_names`.

**Кворумная синхронная репликация.** `ANY 1 (replica_1, replica_2)` означает: COMMIT
подтверждается клиенту только после того, как запись WAL сброшена минимум на одной из
двух реплик. Проверка семантики на тестовой таблице:

| Живых реплик | Поведение COMMIT |
|---|---|
| 2 | подтверждается сразу, у обеих реплик `sync_state = quorum` |
| 1 | подтверждается — кворум `ANY 1` всё ещё собирается |
| 0 | **зависает** (клиент убит по таймауту 6 с), т.к. подтверждать некому |

Состояние репликации на мастере:

```
select application_name, state, sync_state from pg_stat_replication;
 application_name |   state   | sync_state
------------------+-----------+------------
 replica_1        | streaming | quorum
 replica_2        | streaming | quorum
```

Попутный нюанс стенда: при апгрейде на образ postgres:18 пришлось поменять точку
монтирования volume с `/var/lib/postgresql/data` на `/var/lib/postgresql` — 18-е
docker-образы хранят данные в major-версионном подкаталоге (`.../18/docker`).

## 2. Перенос запросов на чтение на слейвы

Выбраны два запроса на чтение из спецификации: **`GET /user/get/{id}`** (в реализации —
`GET /user/{id}`) и **`GET /user/search`**.

Стек — .NET, поэтому вместо `ReplicationRoutingDataSource` из java-примера использован
его прямой аналог из Npgsql — **`NpgsqlMultiHostDataSource`**. Строка подключения
перечисляет все узлы кластера:

```
Host=localhost:5432,localhost:5433,localhost:5434;Database=people_hub;...;Load Balance Hosts=true;
```

Роутинг выполняется на уровне `DbClient`: каждый метод доступа к данным имеет флаг
`readOnly` (аналог `@Transactional(readOnly = true)`), по которому выбирается тип
целевого узла:

```csharp
var connection = dataSource.CreateConnection(readOnly
    ? TargetSessionAttributes.PreferStandby   // чтение — со слейва
    : TargetSessionAttributes.Primary);       // запись — на мастер
```

Npgsql сам определяет, кто из хостов primary, а кто standby, и с
`Load Balance Hosts=true` балансирует read-only подключения между репликами.
`PreferStandby` (а не `Standby`) выбран сознательно: при недоступности всех реплик
чтение деградирует на мастер, а не падает.

Оба выбранных запроса в репозитории выполняются через read-only путь
(`ExecuteDataTableAsync` → `readOnly: true`), записи (`INSERT`/`UPDATE`) идут через
`readOnly: false` → `Primary`.

## 3. План нагрузочного тестирования

Сценарий `load-testing/search.js` (k6) имитирует пользователя:

- поиск `GET /user/search?first_name=&last_name=` со случайными именем/фамилией и
  пагинацией (10 страниц по 50, каждые 5 страниц — новый поисковый запрос);
- по результатам поиска — точечные `GET /user/{id}` (5 штук за итерацию);
- профиль нагрузки: линейный рост до 400 VU за 200 секунд;
- пороги: p95 поиска < 2 c, p95 get-by-id < 1 c.

База: 1 млн сгенерированных пользователей, GIN-индекс
`(surname gin_trgm_ops, name gin_trgm_ops)`.

## 4. Замеры: нагрузка перешла на слейвы

Два прогона одного и того же плана:

- **Baseline** — приложению доступен только мастер (`Host=localhost:5432`);
  отчёт `run1-master-only.html`;
- **Reads → replicas** — полная multi-host строка, чтение через `PreferStandby`;
  отчёт `run2-master-plus-replicas.html`.

Главное свидетельство переноса нагрузки — `docker stats` в середине прогона:

| Узел | Baseline | Reads → реплики |
|---|---|---|
| pg-master | **2381 % CPU** | 15 % CPU |
| pg-replica-1 | 0.00 % | **1283 % CPU** |
| pg-replica-2 | 0.01 % | **1064 % CPU** |

В baseline мастер в одиночку съедает весь CPU и отбивает подключения
(`FATAL: sorry, too many clients already`), реплики простаивают. После переноса чтения
нагрузка распределилась по всем трём узлам, мастер разгрузился более чем вдвое.

Метрики k6:

| Метрика | Baseline (только мастер) | Reads → реплики |
|---|---|---|
| http_reqs | 63 255 (303.2 req/s) | 64 470 (308.7 req/s) |
| ошибки HTTP | 0 % | 0 % |
| `user/{id}` медиана | 161 ms | **53 ms (×3)** |
| `user/search` медиана | 539 ms | **446 ms (×1.2)** |
| p95 общая | 1.18 s | 1.52 s |

Медианная латентность улучшилась в разы — быстрые запросы перестали стоять в очереди к
единственному узлу. Суммарная пропускная способность при этом не выросла, а хвост
(p95/p99) стал длиннее — это ограничение стенда, а не подхода: все три узла делят CPU
одной машины, суммарные ~2400 % CPU были исчерпаны уже в baseline. Репликация
перераспределяет нагрузку, но не добавляет железа; на отдельных хостах ожидаемый эффект —
рост ёмкости чтения ~×3. Подробное сравнение — в `report1.md`.

## 5. Эксперимент: аварийная остановка мастера под write-нагрузкой

Полный отчёт — `failover-report.md`, кратко:

**Нагрузка на запись.** Добавлен API-эндпоинт `POST /user/register` (INSERT в `users`),
скрипт `load-testing/write-load.js` (k6, 20 VU, 240 с) пишет строки с маркером
`city='__failover_test__'` и на своей стороне считает подтверждённые записи
(counter `writes_confirmed` — только ответы 200 с `user_id`).

**Хронология:**

1. t=0 — старт write-нагрузки (латентность записи: медиана 14.8 ms);
2. t=60 c — `docker kill pg-master` (SIGKILL); на этот момент обе реплики на
   LSN `0/147E48D0`;
3. t=60..121 c — записи отваливаются (11 586 отказов), приложение живо, чтение работает;
4. t=121 c — сверка реплик: `pg_last_wal_receive_lsn()` идентичен у обеих (`0/147E7848`) —
   выбираем любую; промоут `pg-replica-1` через `select pg_promote()`;
5. t=121..130 c — переключение второй реплики на новый мастер без перезаливки: на новом мастере создан
   слот `replica_2_slot`, на реплике — `alter system set primary_conninfo = 'host=pg-replica-1 ...'`
   + `pg_reload_conf()` (рестарт не нужен);
6. t=~130..240 c — запись восстановилась, k6 снова копит `writes_confirmed`;
7. t=240 c — конец нагрузки; итог k6: **23 554 подтверждённых** записи;
8. проверка потерь.

**Результат — потерь подтверждённых транзакций нет:**

| Где считали | Строк с маркером |
|---|---|
| k6 (подтверждено клиенту) | **23 554** |
| новый мастер (бывшая replica-1) | **23 554** |
| replica-2 после переключения | **23 554** |

Динамика по времени — в HTML-дашборде `run3-failover.html`: на графике запросов
виден обрыв успешных записей ровно на 60-й секунде и их возврат после промоута на ~130-й.

Это прямое следствие кворумной синхронной репликации: каждая подтверждённая клиенту
транзакция в момент SIGKILL уже была на минимум одной реплике. Обратная сторона
зафиксирована отдельным экспериментом (раздел 1): транзакция, чей клиент оборвался во
время ожидания кворума, может оказаться в базе, не будучи подтверждённой — гарантия
работает только в направлении «подтверждено → не потеряно».

Бонус-наблюдение: приложение с multi-host connection string пережило failover без
перезапуска и изменения конфигурации — после промоута Npgsql заново обнаружил, какой
узел является primary, и запись восстановилась (проверено контрольным
`POST /user/register` после промоута).

## 6. Выводы

1. Потоковая репликация 1×мастер + 2×слейва настроена декларативно в docker-compose
   (слоты, `pg_basebackup -R`, hot standby) и воспроизводится с нуля одной командой.
2. Чтение (`/user/search`, `/user/{id}`) переведено на слейвы средствами Npgsql
   (`NpgsqlMultiHostDataSource` + `TargetSessionAttributes` по флагу read-only) — без
   прокси и внешних балансировщиков.
3. Замеры подтверждают перенос нагрузки: реплики с 0 % поднялись до ~1283 % и ~1064 % CPU,
   мастер разгрузился с 2381 % до 15 %, медианная латентность чтения упала в разы.
4. Кворумная синхронная репликация `ANY 1` под аварийной остановкой мастера (kill -9 под
   write-нагрузкой) не потеряла ни одной из 23 554 подтверждённых транзакций; failover
   (промоут свежайшей реплики + перенацеливание второй) выполняется без перезаливки данных,
   и кворумный режим сохраняется на новом мастере (настройка из `postgresql.auto.conf`
   копируется репликам при basebackup).
